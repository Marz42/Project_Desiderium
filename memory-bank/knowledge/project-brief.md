面向单一管理者的番剧解说选题辅助系统：

> 维护不超过100个频道、账号、关键词和榜单监控项，以YouTube为稳定主数据源，每日识别“多个相关频道集中创作且视频相对频道平均表现更好”的趋势主题，生成约30个候选创作方向，由管理者筛选10—15个并导出简报。

# 一、MVP产品定义

## 1. 目标用户

第一阶段只有一名运营策划兼管理者使用Web后台。

5—10名剪辑师不登录系统，只接收管理者审核后的Markdown或HTML简报。

## 2. 优先解决的问题

优先级已经确定为：

1. 将每日1—3小时的无标准搜索，转化为30—60分钟内完成的结构化选题审核。
2. 提前发现多个频道同时跟进、且表现异常的破圈题材。
3. 为团队每天约25—50条Shorts和5—10条长视频提供可复用的趋势主题和不同内容角度。

系统不需要覆盖团队当天的全部产量。一个持续半个月的趋势可以每天提供新的内容卖点和片段角度。

## 3. MVP明确不做

第一版不做：

* 剪辑师账号和复杂权限；
* 自动分配选题；
* 自动生成完整英文脚本；
* 自动生成英文标题或钩子；
* 视频画面理解；
* 自动下载或管理番剧素材；
* 完整、稳定的TikTok全站数据服务；
* 个性化账号推荐；
* 自动发布；
* 复杂机器学习预测爆款。

# 二、核心业务对象

## 1. Watchlist：统一监控列表

系统不应分别开发“频道表”“关键词表”“榜单表”，而应统一为一个可扩展的监控项模型：

```text
WatchItem
- 类型：频道 / 账号 / 关键词 / 动漫作品 / 榜单页面
- 平台：YouTube / TikTok / 其他
- 名称
- URL或平台ID
- 监控等级：重点 / 一般 / 实验
- 标签
- 备注
- 是否启用
- 最后抓取时间
- 最近抓取状态
- 失败次数
```

MVP支持：

* CSV批量导入；
* 设置重点、一般或实验等级；
* 添加标签和备注；
* 启用、停用；
* 查看最后抓取时间；
* 查看成功、失败和错误信息。

建议的CSV字段：

```csv
type,platform,name,url_or_id,tier,tags,note,enabled
channel,youtube,Anime Recap Channel,UCxxxx,priority,"anime,recap","重点竞品",true
keyword,youtube,anime recap,,general,"generic","通用搜索词",true
account,tiktok,somecreator,@somecreator,experimental,"anime","TikTok实验",true
```

## 2. 趋势主题与创作方向

采用两层结构。

上层是稳定的趋势主题：

```text
动漫作品
→ 角色、篇章或具体事件
→ 主题生命周期
```

下层是每天可执行的创作方向：

```text
你可能忽略的三个细节
角色为什么背叛主角
最残酷的一场战斗
真正的结局
适合Shorts还是长视频
```

同一个趋势主题连续出现15天时，不创建15个重复趋势，而是复用同一个趋势ID，并更新：

* 今日状态；
* 热度变化；
* 新出现的视频；
* 新发现的内容卖点；
* 管理者历史处理记录。

# 三、管理者每天的工作流

系统后台每天自动完成：

```text
采集监控列表
→ 更新视频数据快照
→ 计算单视频异常程度
→ 聚合相似视频为趋势主题
→ 判断趋势上升或下降
→ 生成创作方向
→ 排序并保留约30个候选
```

管理者打开后台后：

1. 查看约30个候选方向；
2. 查看趋势证据和代表视频；
3. 勾选10—15个方向；
4. 添加人工备注；
5. 标记选题状态；
6. 导出Markdown或HTML；
7. 转发给剪辑团队。

目标不是让管理者逐条观看所有视频，而是把审核范围从数百条视频压缩到约30个结构化候选。

# 四、采集策略

## 1. YouTube主流程

YouTube频道采集和关键词发现应分开处理。

### 重点频道：20—30个

重点频道采集：

* 最新发布视频；
* 标题、简介、发布时间、时长和标签；
* 播放、点赞、评论等统计数据；
* 定时统计快照；
* 尝试获取字幕或转录；
* 获取该频道近期视频，用于计算频道自身基准。

建议每4—6小时检查一次重点频道，但每天只生成一次简报。

### 一般频道

一般频道每日发现一次新视频，随后对已经进入候选池的视频继续更新统计快照。

只采集：

* 标题；
* 简介；
* 标签；
* 发布时间；
* 播放、点赞和评论；
* 频道基本信息。

### 关键词

关键词包括：

* 通用词：`anime recap`、`anime explained`；
* 具体作品名；
* 角色名；
* 篇章或剧情词；
* 系统建议、管理者确认的新词。

YouTube官方搜索接口支持关键词、发布时间范围和排序参数；当前官方默认限制为每天100次`search.list`调用，其他多数读取接口共享每日10,000单位，因此关键词需要按“关键词组”合并调度，不能为每个词频繁翻页搜索。([Google for Developers][1])

建议MVP将活跃关键词控制在10—30组，每组每天调用1—2次，把主要采集能力放在固定频道上。

YouTube视频接口能够读取公开视频的基本信息和统计数据，也可以读取按地区划分的热门视频榜单。([Google for Developers][2])

## 2. 字幕与转录

重点监控做到“字幕或转录分析”，但不能依赖YouTube官方字幕接口作为主要方案。

官方`captions.list`需要授权，而且返回的是字幕轨道信息，不直接返回字幕文本。因此它不适合作为任意竞品视频的通用字幕数据源。([Google for Developers][3])

MVP应采用分层降级：

```text
公开字幕可提取
→ 使用公开字幕

没有字幕但视频进入高优先级候选
→ 可选本地或API语音转录

无法获取
→ 只使用标题、简介和元数据分析
```

语音转录只对最终候选或重点频道视频执行，避免不必要的成本和数据处理。

## 3. TikTok实验模块

TikTok作为可插拔实验源，不参与主流程可用性承诺。

官方Display API主要读取经过用户授权的TikTok创作者自身近期视频，并不是全站热门视频搜索接口；Research API明确不向创作者、广告主或商业用户开放。([developers.tiktok.com][4])

因此MVP的TikTok模块采用：

```text
TikTokAdapter
├─ 指定账号抓取
├─ 指定关键词抓取
├─ 公开榜单或标签页抓取
├─ Cookie配置
└─ 独立失败与重试
```

要求：

* TikTok任务失败不影响YouTube简报；
* Cookie单独存储；
* 页面结构和选择器集中配置；
* 所有抓取结果标记数据来源和抓取时间；
* 实验结果可以进入候选池，但默认降低来源置信度；
* 不把TikTok抓取稳定性作为MVP验收条件。

# 五、热度算法

## 1. 先判断单条视频是否“异常”

绝对播放量不能直接比较。头部频道的100万播放可能是常态，中小频道的20万播放可能是重大异常。

对于视频 (i)，定义：

```text
视频速度 = 观察窗口内新增播放量 ÷ 窗口小时数
```

如果还没有历史快照，则冷启动使用：

```text
估算速度 = 当前播放量 ÷ 视频发布小时数
```

频道基准使用该频道最近约20条视频的中位数，而不是平均数：

```text
频道基准速度
= 最近视频在相似发布时长下的播放速度中位数
```

发布时长应至少分成：

* 0—6小时；
* 6—24小时；
* 1—3天；
* 3—7天。

单视频突破倍数：

```text
BreakoutRatio
= 当前视频速度 ÷ 频道同龄视频基准速度
```

初始解释规则：

| BreakoutRatio | 含义       |
| ------------: | -------- |
|           < 1 | 低于频道一般表现 |
|           1—2 | 正常或略好    |
|           2—4 | 明显异常     |
|           > 4 | 强突破信号    |

这里用“频道自身基准”完成账号规模校正。

## 2. 什么叫“热门破圈题材”

MVP初始门槛建议定义为：

```text
72小时内至少3个不同监控频道发布相关内容
并且
至少一半相关视频的BreakoutRatio ≥ 2
```

或者满足更敏感的早期条件：

```text
24小时内至少2个不同频道发布
其中至少1条BreakoutRatio ≥ 4
并且关键词或榜单数据提供额外佐证
```

第一类进入正式趋势候选，第二类标记为“早期信号”。

单一频道的一条爆款视频不能直接被认定为破圈趋势。

## 3. 趋势综合评分

所有分项先在当天候选趋势中转换成0—100百分位，避免不同量纲直接相加。

建议初始公式：

```text
TrendScore =
  35% 跨频道共振
+ 25% 相对表现异常
+ 20% 近期增长动量
+ 10% 七日持续性
+  5% 绝对传播规模
+  5% 新鲜度
```

### 跨频道共振：35%

考虑：

* 24小时、72小时内不同频道数量；
* 重点频道和一般频道覆盖；
* 发布时间是否集中；
* 是否来自不同频道而非重复搬运账号。

### 相对表现异常：25%

考虑：

* 相关视频的BreakoutRatio中位数；
* BreakoutRatio大于2的视频比例；
* 是否存在中小频道异常爆发。

### 近期增长动量：20%

根据统计快照计算：

```text
当前24小时标准化新增播放
÷
前一个24小时标准化新增播放
```

标准化新增播放是将视频新增播放除以频道基准，再按趋势聚合。

### 七日持续性：10%

考虑：

* 过去7天有多少天出现新视频；
* 是否持续有不同频道加入；
* 是否仍有新内容卖点出现。

### 绝对传播规模：5%

用于保留确实覆盖面很广的内容，但权重较低，防止大频道垄断排名。

### 新鲜度：5%

新发现趋势获得轻微加分，但不应压过持续上升的成熟趋势。

## 4. 趋势方向标签

每天计算趋势活动值：

```text
Activity24 = 最近24小时标准化新增播放与新增视频数量
ActivityPrev = 前24小时对应活动值
GrowthRatio = Activity24 ÷ ActivityPrev
```

初始规则：

| 状态   | 判定                       |
| ---- | ------------------------ |
| 新发现  | 首次识别时间不超过24小时            |
| 持续上升 | GrowthRatio ≥ 1.25       |
| 热度稳定 | 0.8 ≤ GrowthRatio < 1.25 |
| 开始下降 | GrowthRatio < 0.8        |
| 重新升温 | 曾经下降，当前GrowthRatio ≥ 1.5 |

这些阈值必须配置化。运行两到四周后，根据人工判断结果校准，不能把首版阈值视为永久真值。

## 5. 三时间窗口

系统同时保存：

* 24小时：发现新视频和突发变化；
* 72小时：判断跨频道共振和短期方向；
* 7天：判断持续性、重复和生命周期。

每日只生成一次简报，但底层指标采集应至少每4—6小时运行一次。否则无法真正计算增长速度，只能比较静态播放量。

# 六、趋势聚类与去重

趋势聚类采用“确定性规则优先，LLM处理歧义”的方式。

第一层先使用：

* 动漫作品别名表；
* 英文名、日文罗马字和常用缩写；
* 角色名；
* 篇章名；
* 标题关键词；
* Watchlist标签。

第二层使用文本向量相似度聚类。

第三层只对难以判断的候选调用LLM，例如：

* 两个标题是否在讨论同一剧情；
* 某个角色话题是否属于同一个趋势；
* 两个内容卖点应该合并还是拆开。

系统内部应分别保存：

```text
TrendTheme
- 作品
- 角色、篇章或剧情事件
- 生命周期状态

CreativeAngle
- 内容卖点
- 建议形式
- Short / Long / Both
```

同一趋势下可以每天产生新的CreativeAngle，但应与以下内容做去重：

* 最近7天已推荐方向；
* 管理者标记已采用的方向；
* 已发布视频标题和链接；
* 被标记“不再推荐”的方向。

---

# 七、LLM负责的部分

LLM只处理语言和语义任务：

* 将多个视频归纳为一个趋势；
* 中文解释为什么热门；
* 从字幕、标题或简介中提取内容卖点；
* 生成中文创作方向；
* 判断适合Shorts、长视频或两者；
* 翻译英文原标题；
* 解释趋势评分证据。

建议结构化输出：

```json
{
  "trend_name_zh": "",
  "anime_title": "",
  "topic_entities": [],
  "why_trending_zh": "",
  "creative_angles": [
    {
      "angle_zh": "",
      "format": "short|long|both",
      "evidence_video_ids": []
    }
  ],
  "confidence": 0.0
}
```

LLM不得负责：

* 编造播放数据；
* 判断增长速度；
* 直接计算评分；
* 无证据地声称“多个频道都在做”；
* 单独决定趋势是否上升；
* 修改原始数据。

# 十、选题状态管理

CreativeAngle至少支持：

```text
candidate       候选
selected        进入简报
adopted         已采用
published       已发布
reusable        可换角度继续做
blocked         不再推荐
```

进入`published`时填写：

* 发布链接；
* 发布日期；
* 可选备注。

MVP暂不自动获取团队账号数据，但数据模型要预留：

```text
published_views
published_likes
published_comments
```

后续可以接入团队YouTube账号，验证系统推荐是否真正提升表现。

# 十一、技术架构

## MVP技术栈

```text
Python
FastAPI
Jinja2 + HTMX
PostgreSQL
SQLAlchemy + Alembic
APScheduler或系统Cron
Docker Compose
```

选择服务端渲染而不是React，可以明显减少单管理者后台的开发量。

任务结构：

```text
Web Process
├─ Watchlist管理
├─ 候选审核
├─ 状态记录
└─ 导出

Scheduler
├─ YouTube频道采集
├─ YouTube关键词搜索
├─ 视频统计快照
├─ 字幕或转录任务
├─ TikTok实验任务
└─ 每日趋势生成

Analysis Pipeline
├─ 数据标准化
├─ 频道基准
├─ 视频异常评分
├─ 趋势聚类
├─ 趋势评分
└─ LLM分析
```

## 上位替代

| MVP方案            | 上位替代                       |
| ---------------- | -------------------------- |
| APScheduler/Cron | Celery、Temporal或独立工作流引擎    |
| Jinja2 + HTMX    | React或Next.js              |
| PostgreSQL普通表    | TimescaleDB或ClickHouse时间序列 |
| Python内存向量计算     | pgvector、Qdrant或其他向量库      |
| 本地语音转录           | 托管ASR服务                    |
| TikTok实验抓取       | 商业数据供应商                    |
| HTML/Markdown导出  | 飞书、Discord或Slack自动推送       |
| 单账号              | RBAC、多团队和审批流程              |

对于100个以内的监控项，不需要在MVP使用Kafka、Elasticsearch、独立向量数据库或Kubernetes。

# 十四、MVP验收标准

## 业务验收

* 每日选题搜索和判断时间从1—3小时降到30—60分钟；
* 每天生成约30个候选；
* 管理者可以选出至少10个可执行方向；
* 前15个候选中至少60%被认为有参考价值；
* 能够支持同一持续趋势产生多个不同内容卖点；
* 管理者可以在一小时内完成审核和导出。

## 算法验收

* 能识别至少3个频道集中制作的趋势；
* 能显示视频相对各自频道基准的表现；
* 能识别中小频道异常突破；
* 能区分新发现、上升、稳定和下降；
* 所有评分都有原始数据和公式依据；
* 趋势聚类错误可以追踪和纠正。

## 工程验收

* YouTube主流程连续稳定运行；
* 重点频道数据新鲜度控制在约6小时；
* 一般频道数据新鲜度控制在24小时；
* TikTok失败不影响YouTube；
* 所有抓取任务有日志；
* 支持本地Docker运行；
* 可以迁移到VPS或长期在线电脑；
* Markdown和HTML可以正常导出。

# 十五、关键设计结论

第一，系统的核心不是LLM，而是Watchlist、统计快照、频道基准和趋势生命周期。

第二，“热门破圈”的主要判据必须是：

> 跨频道共振 × 相对频道平均表现异常。

第三，每天只生成一次简报，不代表每天只采集一次。底层至少需要数次统计快照，才能判断趋势方向。

第四，趋势主题必须跨日持续存在。每天变化的是评分、状态、代表视频和创作方向，而不是每天重新生成一组无关联的热点。

第五，先用YouTube完成可验证闭环。TikTok只能作为隔离的实验适配器，不能成为MVP主路径。

第六，最先开发的不是完整Web界面，而是“影子验证”：用真实频道和一段时间的数据验证BreakoutRatio、跨频道共振和趋势排序。评分机制通过后，再建设管理后台。
